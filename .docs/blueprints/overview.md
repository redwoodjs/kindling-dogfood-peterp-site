# 2000ft View
This repository is a very small static site fork used as a dogfood target for Kindling. It publishes a personal homepage for Peter Pistorius and serves primarily as a lightweight GitHub Pages-style deployment target rather than a full application. The main user is the repository maintainer or automation acting on their behalf, and the current development focus appears to be the delivery pipeline and validation path rather than site content.

# System Flow
1. The only visible site entry point is [`index.html`](index.html), which contains the full HTML document and inline styles.
2. The repository currently has no package manifest, build script, or app source tree, so there is no intermediate compile step on the visible code path.
3. GitHub-facing metadata lives in [`.github/PULL_REQUEST_TEMPLATE.md`](.github/PULL_REQUEST_TEMPLATE.md); there are no checked-in workflows yet.
4. Kindling task state lives under [`.kindling/`](.kindling/), which is outside the product runtime but relevant to how this repo is being operated.
5. The expected future delivery path for the migration task is likely GitHub Actions plus whatever static hosting target the repo uses, but that machinery is not present in the current tree.

# Directory Map
- [`.github/`](.github/): GitHub metadata; currently only a pull request template is present.
- [`.kindling/`](.kindling/): Local task state and progress logs generated by Kindling runs.

# Key Abstractions
- `index.html`: The entire user-facing site is defined in one HTML file. It includes the page title, inline CSS, and the homepage content in the body. There is no separation between content, styling, and behavior.
- PR template: The repository uses a pull request template to remind contributors that this fork is the active Kindling dogfood target and that pull requests should target this fork’s `master` branch rather than upstream.
- Kindling task state: The `.kindling` directory stores the current task metadata and progress log. It is part of the harness workflow, not the deployed site, but it explains why this repository may contain automation-generated changes.

# Conventions Observed
- The site is implemented as a single standalone HTML document.
- Styling is inline in the page head rather than split into separate assets.
- There is no build tooling, dependency manifest, or test suite visible in the repository root.
- The repository is explicitly treated as a Kindling dogfood fork in both the README and PR template.
- There are no checked-in GitHub Actions workflows in `.github/` yet.
