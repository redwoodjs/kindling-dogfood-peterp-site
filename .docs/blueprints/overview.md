# 2000ft View
This repository is the kindling dogfood fork of peterp.org. Today it is a very small static personal site that serves a single HTML page, and the active migration epic is to rewrite that site as a RedwoodSDK application that runs on Cloudflare Workers. The site is for Peter Pistorius, and the epic scope is the framework migration itself, not the domain transfer or traffic cutover.

# System Flow
1. The current entry point is `index.html` at the repo root.
2. The browser loads the document, which sets the page title, basic meta tags, and a small inline style block.
3. The body renders the profile introduction, social links, and a short list of side projects directly in static markup.
4. There is no application runtime, build pipeline, or server entry point in the repository yet.
5. The migration issue defines the future shape: a RedwoodSDK app with routed components, worker configuration, CI deploy workflow, and local verification.

# Directory Map
- `./` Root-level static site files and repo metadata.
- `.github/` GitHub workflow and pull request template metadata.
- `.kindling/` Local kindling task state and harness files.
- `.docs/` Documentation area; `.docs/blueprints/` does not exist yet in the current tree and is the intended location for this blueprint artifact.

# Key Abstractions
The `index.html` document is the only application surface today. It combines structure, content, and styling in one file, which makes it the current source of truth for all visible site behavior.

The `README.md` frames the repository as a kindling dogfood fork of `peterp/peterp.github.io`. It explains that pull requests, issues, and commits here are expected to be generated by kindling tasks rather than by normal manual maintenance.

The migration epic in GitHub issue `#8` is the main architectural guide for the repo’s next state. It defines the target as RedwoodSDK on Cloudflare Workers and breaks the work into scaffold, porting content, porting styles, local verification, worker config, CI deploy, and GitHub Actions validation.

The `.github/PULL_REQUEST_TEMPLATE.md` is a repository-level convention artifact. It clarifies that pull requests in this fork should target this fork’s `master`, not the upstream peterp repository.

# Conventions Observed
- The site is currently implemented as a single file rather than a component tree.
- Styling is inline in `index.html` rather than split into separate assets.
- Repository metadata is minimal; there is no `package.json`, lockfile, or framework scaffolding in the current tree.
- The project uses GitHub Issues for work tracking, and the epic explicitly decomposes the migration into numbered sub-issues.
- The repository is set up as a dogfood fork, so references to upstream are part of the normal project vocabulary.

# Known Unknowns
- The eventual RedwoodSDK project structure is not yet present, so the future source tree, routing layout, and component boundaries are unknown.
- The build and local development commands are not defined in the repository yet.
- Worker configuration is not present, so deployment shape and Cloudflare-specific settings are still unknown.
- CI workflow details are not present beyond the request for a GitHub Actions deploy pipeline.
- Testing strategy is not visible in the current tree.
- It is unclear whether the migration will preserve the current page content verbatim or adapt it during the port.
