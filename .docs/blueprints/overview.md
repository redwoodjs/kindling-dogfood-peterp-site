# kindling-dogfood-peterp-site — Blueprint

### 2000ft View

`redwoodjs/kindling-dogfood-peterp-site` is a fork of Peter Pistorius's personal static site (`peterp.github.io`), repurposed as a live dogfood target for the RedwoodJS *kindling* agentic task harness. It is a deliberately simple GitHub Pages site whose issues, PRs, and commits are generated by kindling tasks. The repo contains a single static HTML page and minimal CI, serving as a sandbox where kindling can exercise real-world automation — adding dependencies, editing workflows, and opening PRs — without risk to production code.

### System Flow

1. **Kindling harness** creates a task file in `.kindling/board/doing/` with a brief and checklist.
2. An **agent** reads the repo, executes the task, and produces changes.
3. Changes are committed and pushed to a branch; a **PR** is opened against `master` on this fork.
4. **GitHub Actions** runs `.github/workflows/ci.yml` on pull requests and pushes to `master`. Currently this is a placeholder job that does nothing.
5. **GitHub Pages** serves `index.html` at the domain configured in `CNAME` (`peterp.org`).

### Directory Map

- `.github/` — GitHub metadata: PR template and workflow definitions.
- `.github/workflows/` — GitHub Actions CI configuration.
- `.kindling/` — Kindling harness metadata.
- `.kindling/board/` — Kanban-style task board with directories for each task lifecycle stage (`inbox`, `todo`, `doing`, `in-review`, `done`, `archived`, `dormant`).
- `.docs/blueprints/` — Architectural context for future agent tasks.

### Key Abstractions

- **Kindling Board** — The `.kindling/board/` directory acts as a lightweight kanban. Task files migrate between subdirectories (`inbox` → `todo` → `doing` → `done`) to track state. Each file is a markdown document with YAML front matter containing status, labels, timestamps, and GitHub cross-references.
- **Task File** — A markdown file in the board directory that defines a kindling task. Front matter fields include `status`, `github-pr`, `depends-on`, and `no-pr`. The body contains a brief and an optional checklist/progress log.
- **Placeholder CI** — `.github/workflows/ci.yml` is the only automation running in GitHub. It is intentionally minimal (`run: "true"`) and exists so that kindling tasks have a workflow to iterate on.
- **Static Site** — The "application" itself is a single `index.html` file with no build step. It is served by GitHub Pages.

### Conventions Observed

- Task files are named with an ISO date prefix and a slugified brief, stored in `.kindling/board/<stage>/`.
- Commits made by kindling agents include `[skip ci]` in the message when appropriate.
- The PR template reminds contributors that PRs target this fork's `master`, not upstream.
- There is no existing package manager configuration in the working tree (no `package.json`, `pnpm-lock.yaml`, etc.).
