### 2000ft View

This repository is a kindling dogfood fork of Peter Pistorius's personal website (`peterp.org`), maintained by the RedwoodJS team as a live test subject for the [kindling](https://github.com/redwoodjs/kindling) agentic harness. Real humans do not contribute here; pull requests, issues, and commits are produced by kindling agent tasks. The canonical upstream site lives at `peterp/peterp.github.io`. The project is currently mid-migration: the existing site is a single static `index.html`, and the active work is converting it to a [RedwoodSDK](https://docs.rwsdk.com) application — a Vite + React + RSC framework deployed to Cloudflare Workers.

---

### System Flow

**Current (pre-migration):** There is no build step. GitHub Pages serves `index.html` directly. The file contains inline HTML, a small inline `<style>` block, and links to Twitter/GitHub/project pages.

**Target (post-migration):** A RedwoodSDK app scaffolded under the repository root (or a `web/` subfolder). The entry point is `src/worker.tsx`, which exports a `defineApp` function that routes incoming HTTP requests. React components in `src/` render page content. Vite builds the app; `wrangler` deploys the output to a Cloudflare Worker. GitHub Actions triggers deployment on every push to `master`.

Migration path (per the epic sub-issues, in order):
1. Scaffold rwsdk project via `pnpx create-rwsdk` — produces `src/worker.tsx`, `package.json`, `wrangler.toml`, and `pnpm-lock.yaml`.
2. Port `index.html` body content into an rwsdk route component.
3. Move inline CSS into the rwsdk app (global stylesheet or CSS module).
4. Verify `pnpm dev` boots and serves correctly.
5. Confirm `wrangler.toml` reflects the correct worker name and compatibility date.
6. Replace the stub CI workflow with a real deploy workflow (`pnpm release` → Cloudflare Workers).
7. Install `@redwoodjs/agent-ci` so future agents can validate the CI workflow locally before pushing.

### Key Abstractions

**`index.html`** — The complete current site. A single static HTML file with an inline `<style>` block setting `max-width: 480px`, a bio paragraph, social links, and an ordered list of side projects. This is the content and style source of truth for the migration.

**Kindling board** (`.kindling/board/`) — A file-system task board used by the harness. Each task is a markdown file with YAML frontmatter (`status`, `started`, `completed`, `github-pr`, `no-pr`, `depends-on`) and freeform `## Brief`, `## Checklist`, and `## Progress Log` sections. Files live in status-named subdirectories. The harness moves files between directories as task state changes.

**RedwoodSDK `src/worker.tsx`** — After scaffolding, this file becomes the framework entry point. It exports a `defineApp` function that maps URL routes to React components. This is where all routing logic will live.

**`wrangler.toml`** — Cloudflare Workers deployment configuration generated by the rwsdk scaffold. Specifies the worker name, compatibility date, and any bindings. Must be committed and kept aligned with the actual worker in Cloudflare.

**Epic orchestration issue (#48)** — A GitHub issue used as a shared progress board for the full migration. Sub-task checkboxes map 1:1 to numbered issues (#1–#7). The kindling harness updates this issue as tasks complete.

### Conventions Observed

- Package manager is `pnpm` exclusively — `pnpm-lock.yaml` is the expected lockfile; `package-lock.json` and `yarn.lock` are prohibited.
- All pull requests target this fork's `master` branch, not the upstream `peterp/peterp.github.io`.
- The `.github/PULL_REQUEST_TEMPLATE.md` reminds contributors of this fork-targeting rule.
- Kindling task cards use ISO 8601 timestamps in frontmatter and progress log entries.
- The `no-pr: true` frontmatter flag marks tasks that complete without opening a pull request.
- The `github-comments: true` flag marks tasks that post progress to a GitHub issue.
- Issues are cross-referenced with an upstream repo (`peterp/peterp.github.io`) via "Upstream:" links in issue bodies.
- Agent tooling notes (e.g., references to agent-ci) are expected to live in `CLAUDE.md` or `AGENTS.md` at the root.

### Known Unknowns

- **Cloudflare account details** — Worker name, account ID, and `CLOUDFLARE_API_TOKEN` secret configuration are not documented in the repo; they will need to be supplied or created at deploy time.
- **Scaffold target directory** — The epic mentions "into a `web/` subfolder or a fresh branch" but does not commit to either; the scaffolding sub-issue (#1) leaves this open.
- **`pnpm create rwsdk` vs `pnpx create-rwsdk`** — The epic body uses `pnpm create rwsdk`; the rwsdk docs use `pnpx create-rwsdk`. These may differ in behavior; the correct invocation should be confirmed during issue #1.
- **Existing CI workflow fate** — It is unclear whether the stub `ci.yml` will be replaced entirely by issue #6 or extended alongside it.
- **Testing strategy** — No tests exist and no test runner is referenced anywhere in the issue set.
- **RSC / server functions usage** — The epic mentions RSC and server functions as future possibilities but no concrete plan is scoped here.
