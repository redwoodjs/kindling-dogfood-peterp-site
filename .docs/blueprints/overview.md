# 2000ft View
This repository now has a RedwoodSDK starter shell in place for the peterp.org migration. The original static homepage content has been replaced by the generated app structure, and the project is wired for pnpm with a lockfile and starter entry points. The next work will build the actual site content and configuration on top of this framework base.

# System Flow
1. The app is now bootstrapped through the RedwoodSDK starter entry points rather than a single static HTML document.
2. The project has a Vite-based build and runtime shell generated by the RedwoodSDK scaffold.
3. The package manager wiring is pnpm, with a committed lockfile and package metadata aligned to it.
4. GitHub metadata is still limited to the existing pull request template under `.github`, which only warns that PRs here should target this fork rather than upstream.

# Directory Map
- `.github/` contains repository workflow metadata; currently only a pull request template.
- `.kindling/` is present locally during this run and appears to be task state, not product code.
- `README.md` now describes the RedwoodSDK starter and pnpm workflow.
- `package.json` and `pnpm-lock.yaml` define the starter app and its dependencies.
- `src/` contains the generated RedwoodSDK app shell and starter pages.
- `wrangler.jsonc` and `vite.config.mts` provide the runtime and build configuration.
- `CNAME` remains part of the repository identity for the custom domain deployment.

# Key Abstractions
The current codebase now centers on the RedwoodSDK scaffold rather than a single static HTML document. The starter app includes generated page modules, shared link definitions, worker and client entry points, and app-level document and header wiring.

The repository still has a fork boundary. The README and PR template continue to emphasize that this repo is a dogfood fork and that work here targets the fork itself, not the upstream `peterp/peterp.github.io` repository.

The `CNAME` file is part of the deployment identity of the site. It indicates that the repository is wired for a custom-domain deployment identity, even though the app shell now uses RedwoodSDK instead of a static Pages document.

# Conventions Observed
- The site is no longer a one-file static page.

- The repo has package manifest, build configuration, source tree, and lockfile wiring in place.
- Repository guidance is documented in plain markdown files at the root and under `.github`.
