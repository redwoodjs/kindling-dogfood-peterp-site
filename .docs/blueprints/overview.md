# Architecture Blueprint: Overview

## 2000ft View

This repository is a kindling dogfood fork of peterp's personal site (peterp.github.io), used as a live test subject for the kindling agentic harness. The project is in active migration from a static single-file HTML site to a RedwoodSDK application — a Vite-based React framework that runs on Cloudflare Workers. Pull requests, issues, and commits in this repo are produced by kindling tasks in the course of dogfooding the agent. The upstream canonical site lives at peterp/peterp.github.io.

## System Flow

In the pre-migration state (current), there is no build pipeline. The site is served directly by GitHub Pages from the repository root. The entry point is `index.html`.

Post-migration (the target state tracked by epic #8), the flow will be:

1. Source lives under a scaffold structure generated by `pnpm create rwsdk@latest`
2. Vite compiles React components (including React Server Components) targeting Cloudflare Workers
3. `wrangler` deploys the compiled worker bundle to Cloudflare
4. GitHub Actions CI runs on push/PR, eventually deploying to Cloudflare Workers

The scaffold phase (sub-issue #1) establishes the skeleton that all subsequent sub-issues build on.

## Directory Map

```

├── index.html          — current static site entry point (pre-migration)
├── CNAME               — domain binding for GitHub Pages (peterp.org)
├── README.md           — project description noting kindling dogfood purpose
├── .github/
│   └── workflows/
│       └── ci.yml      — placeholder CI workflow (always passes)
└── .kindling/          — kindling harness metadata
```

After scaffolding (post sub-issue #1), a new subdirectory or restructured root will contain the rwsdk application tree. The exact output shape depends on the scaffold generator.

## Key Abstractions

**The static index.html** is the current entire site — a single HTML file containing Peter's personal page content: bio, social links, and a list of side projects. It is the source of content that sub-issues #2 and #3 will port into rwsdk routes and components.

**RedwoodSDK (rwsdk)** is the target framework — a Vite-based React meta-framework from the RedwoodJS team, designed to run on Cloudflare Workers. It uses React Server Components and server functions. The scaffold command is `pnpm create rwsdk@latest`. The dev server runs on port 5173 by default (Vite default).

**Cloudflare Workers** is the target runtime. The compiled application bundle runs as a Worker. Configuration lives in `wrangler.toml` (added in sub-issue #5).

**GitHub Pages** is the current hosting mechanism. The `CNAME` file and static `index.html` are served directly from the repository root. This will be superseded by Cloudflare Workers hosting once the migration is complete, but the domain and hosting cutover are tracked separately (out of scope for this epic).

**Kindling CI** — the `.github/workflows/ci.yml` is a placeholder that always passes. A real CI/CD workflow targeting Cloudflare deployment is added in sub-issue #6.

## Conventions Observed

- `pnpm` is the required package manager for all operations — no npm or yarn.
- The CI workflow uses a minimal placeholder pattern (`run: "true"`) to keep PRs green during the scaffolding and migration phases before real CI is wired up.
- Commits from kindling tasks include `[skip ci]` in the message where appropriate (observed in recent git log).
- PR descriptions use `## Context` + `## Details` for scaffold/migration framing; `## Problem` + `## Solution` is reserved for bug fixes.

## Known Unknowns

- The exact scaffold output shape of `pnpm create rwsdk@latest` is not known ahead of running — whether it scaffolds into the current directory or requires a subdirectory argument, and whether non-interactive flags exist, must be determined from the docs at https://docs.rwsdk.com/getting-started/quick-start/.
- Whether the `CNAME` file and `index.html` should be removed or preserved alongside the scaffold output is not specified in the sub-issue — likely to be resolved by the porter (sub-issues #2/#3).
- The `wrangler.toml` configuration details (account ID, worker name, routes) are deferred to sub-issue #5.
- Build and deploy verification specifics (Cloudflare account credentials, deployment targets) are deferred to sub-issues #6 and #7.
