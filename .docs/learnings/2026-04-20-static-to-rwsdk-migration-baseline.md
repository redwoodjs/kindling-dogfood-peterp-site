# Learnings: Static-to-rwsdk Migration Baseline Investigation

**Date:** 2026-04-20  
**Context:** Investigation phase for Epic #8 (peterp.org → RedwoodSDK migration)

Two non-obvious findings from this investigation that future agents should know before touching this codebase.

---

## 1. Inherited HTML bugs must be fixed during porting, not copied

When porting content from a static HTML file into React components, assume the source HTML may contain bugs. In this case, the Machinen project entry (`index.html` line 33) has a malformed anchor:

```html
<a href="https://machinen.dev"></a><i>Machinen</i></a>
```

The opening `<a>` wraps an empty element; the visible text `Machinen` is a sibling node, not a child of the link. Copying this structure into JSX would reproduce the bug — the text would render but the link would be invisible/empty.

**The correct port wraps the text inside the anchor:**

```jsx
<a href="https://machinen.dev"><i>Machinen</i></a>
```

**Lesson:** Before porting HTML content mechanically, read the source for structural correctness. A content parity task is not just a copy — it is an opportunity (and obligation) to fix pre-existing markup errors.

---

## 2. CI deploy workflows require out-of-band human action for cloud secrets

The CI deploy sub-issue (#6) cannot complete end-to-end without a `CLOUDFLARE_API_TOKEN` secret present in the GitHub repository settings. This secret cannot be committed to the repo, cannot be generated by a code change, and cannot be added by an agent — it requires a human to log into GitHub, navigate to repository secrets, and add the token manually.

**Implication for task sequencing:** The deploy workflow file can be authored and merged by an agent. But any CI run that attempts a live deploy will fail until the secret is present. Do not treat a failing deploy CI run as a code bug until the secret has been confirmed to exist.

**Lesson:** When planning infra tasks that involve cloud credentials, explicitly flag the out-of-band human dependency as a prerequisite. The task brief for sub-issue #6 should document this so the agent handling it sets the correct expectation rather than attempting to debug a credential error.
